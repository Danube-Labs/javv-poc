## MAJOR

### M-1. The per-doc `last_scan_order` guard from the contract/INDEX-MAP is not implemented ‚Äî the watermark check is a TOCTOU gate, not an atomic guard. [Axis 1 + 2 ‚Äî drift CONFIRMED; bug PLAUSIBLE]

- CORRECTNESS-CONTRACT ¬ß3 and INDEX-MAP_v4 (findings section, and the `last_scan_order` field comment: *"create/update no-op if scan_order ‚â§ this OR < digest watermark"*) require **both** guards. The code implements only the watermark: `merge_action` (`backend/src/backend/services/merge.py:41-49`) emits an unconditional partial-doc update with no scripted `scan_order > ctx._source.last_scan_order` check.
- **Failing scenario:** envelope A (order 5) passes `advance_watermark` (watermark 4‚Üí5, `services/ingest.py:143`) but its request stalls before the merge bulk (`ingest.py:158`). Envelope B (order 6) then passes the watermark (5‚Üí6), merges, and reconciles (retiring finding X). A's delayed merge now lands last: it re-creates/overwrites X with `present=true`, `last_scan_order=5` ‚Äî exactly the resurrection D40 exists to kill. The watermark check-then-write is not atomic with the cache write; only a per-doc guard closes this.
- **Mitigations in practice:** CronJob `Forbid` + MVP single-pod backend make inverted in-flight envelopes for one digest rare (needs a hung/retried request straddling a cycle). But keystone test 5 promises "final state == newer scan **regardless of arrival order**" and the code doesn't deliver that under interleaving; the test (`test_watermarks.py:150-165`) only races `advance_watermark` itself and asserts on the *watermark*, not on findings after full concurrent `ingest_envelope` calls ‚Äî so the hole is untested (see T-1).
- **Fix direction:** make the findings write a scripted upsert that no-ops when `params.scan_order <= ctx._source.last_scan_order`.

### M-2. Staleness sweep treats every `system-tokens` doc as the scanner registry ‚Äî disabled/rotated/secondary tokens mass-stale a healthy scanner. [Axis 2 ‚Äî CONFIRMED by code; on PR #108, unmerged]

- `run_staleness_sweep` (branch `feat/m3-staleness-sweep`, `backend/src/backend/jobs/staleness.py`, the `match_all` token loop) iterates **all** token docs with no `disabled` filter and no dedup per `(cluster_id, scanner)`.
- **Failing scenario:** token rotation ‚Äî old token disabled 10 days ago (`last_ingest_at` 10d stale), new token ingesting hourly. The sweep processes the old doc: `silent ‚â• M` ‚Üí mass-stales **every** present finding for `(cluster, trivy)`. Whether the new token's healthy-branch revert repairs it in the same sweep depends on arbitrary hit ordering; worst case the whole grid reads `stale` for a day, and `pre_stale_status` churns daily thereafter. Same failure with two *active* tokens where only one is used.
- **Fix direction:** skip `disabled` tokens and reduce to `max(last_ingest_at)` per `(cluster_id, scanner)` before branching.

### M-3. Timezone fragility can crash the whole sweep: `now(UTC) - naive_datetime` raises `TypeError`. [Axis 2 ‚Äî PLAUSIBLE, reachable with a schema-valid envelope]

- `IngestEnvelope.last_seen_at: datetime` (`envelope.py:143`) accepts tz-**naive** ISO strings; the ingest route writes it verbatim into `system-tokens.last_ingest_at` (`routers/ingest.py:129-133`); `_parse_dt` (`staleness.py`) preserves naiveness; then `silent = now - last_ingest` throws for the aware/naive mix ‚Äî one bad-clock scanner kills the staleness sweep **for all tenants**. The JAVV-built scanner sends aware timestamps, so this needs a misbehaving-but-valid client; still, enforce tz-awareness in the envelope validator or coerce in `_parse_dt`.
- **Related (same line):** `last_ingest_at` is **client-supplied time**, not server time ‚Äî a scanner with a future clock permanently suppresses scanner-down escalation. Server-side `datetime.now(UTC)` is the right stamp for an anti-silence guard.

### M-4. `bulk_write`'s 429/503 backoff is a no-op in production. [Axis 3 ‚Äî CONFIRMED; pre-dates M3 but M3's write path depends on it]

- `repositories/bulk.py:22` defines `_noop_sleep` with the comment "replaced in prod wiring" ‚Äî but no caller ever injects a real sleep: `ingest_envelope` calls `bulk_write(client, ‚Ä¶)` bare (`services/ingest.py:134,136,158`). Retries fire back-to-back with zero delay, hammering an already-overloaded cluster. This is the project's own day-one rule ("backoff on 429/503 ‚Äî the only flow control without a broker"). One-line fix: default `sleep=asyncio.sleep`.

## MINOR

- **m-1. SPEC FR-6 says the freshness window is "per-cluster configurable"; the implementation is one global `StalenessTimers` doc** (`staleness.py` `STALENESS_KEY = "staleness"`, no `cluster_id` dimension). Reasonable MVP simplification, but it's an undocumented deviation ‚Äî record it (SPEC note or bolt README Updates) or key the config per cluster. [Axis 1/5]
- **m-2. Per-envelope refresh storm.** `reconcile_absent` forces a full `findings` index refresh per envelope (`services/reconcile.py:65`), plus `refresh:true` on every watermark CAS write and the UBQ. The refresh-before-UBQ is **correct** (it's what prevents a just-merged finding being reconciled off its own scan, and the `last_scan_order < scan_order` filter handles the rest) ‚Äî but at fleet scale this is hundreds of forced refreshes per cycle on the hottest index. Fine for POC; worth a measurement note before M6 read load. [Axis 2 perf]
- **m-3. Token `expiry` is mapped** (`bootstrap.py:102`) and documented in INDEX-MAP ("lifecycle + revocable") **but never enforced** ‚Äî neither `core/auth.py:require_token` nor the ingest inline auth checks it. If enforcement is M5a's, that deferral isn't written anywhere. [Axis 3/6]
- **m-4. Unbounded rate-limiter dict.** `_hits` (`routers/ingest.py:40`) keys on the hash of whatever bearer token is presented; unauthenticated garbage tokens each add a permanent deque. Slow memory leak / cheap DoS vector. Evict empty deques. [Axis 3]
- **m-5. INDEX-MAP internal staleness:** the `findings` section prose still describes reconcile as flipping findings whose "`last_scan_run_id` ‚âÝ the new run" ‚Äî the D40-corrected rule (and the implementation, correctly) uses `last_scan_order <`. Fix the sentence in the source-of-truth doc. The **code** is right. [Axis 1 ‚Äî doc-side drift]
- **m-6. PR #109 wording inconsistency:** PLAN_v4's milestone table says "rebuild-state job ‚Üí **M8a**" (both rows) while the M5c README says M5c **creates** `jobs/rebuild_state.py` and M8a extends it. The M5c-creates/M8a-extends story (and M5c-before-M8a ordering) is sound and consistently told in the bolt READMEs + CORRECTNESS-CONTRACT; only the PLAN table rows omit M5c. One-word fix. [Axis 6]
- **m-7. The "scanner silent since T‚Ä≤" banner has no explicit owner.** SPEC FR-6 requires it on every inventory view; PR #108 correctly frames it as read-time-from-`last_ingest_at` (not written by the sweep ‚Äî good), and M9e's README only mentions a "preview of banner behavior" in the timer editor. No M9a/M9b/M9c README lists the banner itself as a deliverable. Risk of it falling through the M9 split. [Axis 5/6]
