## Verdict

**Healthy, no blockers.** The security-critical M5a surface is well-built and consistently wired:
argon2id + `DUMMY_HASH` timing equalizer, peppered + domain-separated session hashes, the
always-applied tenant filter (structurally un-escapable via the request body), the standing
RBAC/IDOR registry with a build-failing route-table presence check, and the `must_change` gate are
all real, not aspirational. The M3 audit's M-1 fix is present. Four Majors, all
low-likelihood-but-real correctness gaps (the same class as the M-1 the M3 self-review dismissed),
plus minors. Fast-follow the Majors on `main`; none block M5c.

## Major

- **M-1 — a `vex_justification` correction on an already-`not_affected` finding is silently
  dropped.** `triage/service.py` gates state writes on `patch.state != current_state`; re-sending
  `not_affected` with a corrected CISA justification equals the current state, so nothing is
  written or journaled — 200 OK, wrong justification persists in exports. **Fix:** on
  `not_affected → not_affected` with a differing justification, write + journal it.
- **M-2 — decision revoke/edit is check-then-act, no CAS.** `decisions/lifecycle.py` reads
  `revoked_at`, checks in Python, updates without `if_seq_no`/`if_primary_term`. Concurrent
  revokes overwrite the one post-hoc stamp (corrupts past-T reconstruction); concurrent edits
  create two successors → two active decisions forever, breaking the pairing invariant M5c's
  projection needs. **Fix:** CAS the revoke (409 → "already revoked").
- **M-3 — D17 completeness is not guaranteed on partial failure** *(both agents)*. `apply_triage`
  applies the CAS write first, then appends audit rows (which raise). A crash/OS-error after the
  write leaves an applied-but-unjournaled change; the client retry re-reads, sees the target
  already applied (`patch.state == current_state`), no-ops, and **never journals the orphaned
  change**. The writer docstring only reasons about the duplicate-row direction, not the lost-row
  one. **Fix:** journal-before-commit (replay tolerates an orphan row for a CAS that then loses),
  or re-emit on retry-no-diff; at minimum document the window.
- **M-4 — append-series idempotency holds only within one backing index.** Ingest writes
  scan-events/images with a plain `index` action (deterministic `_id`) to the write alias. After
  the lifecycle sweep rolls `-000001 → -000002`, a retried envelope for a pre-rollover scan lands
  in the new index and creates a **second doc with the same `_id`/`commit_key`/`scan_order`**;
  wildcard reads match both, so sum-over-time trend aggregations double-count. Needs a scanner
  retry straddling a monthly rollover — rare. **Fix:** dedup catalog/trend reads by `commit_key`,
  and add a re-push-after-rollover test.

## Minor

- **m-1** — lockout map is unbounded *within* the window (`auth/lockout.py`): a unique-username
  spray grows `_fails` at line rate; `_MAX_KEYS` is a sweep trigger, not a cap. Hard-cap + evict.
- **m-2** — `revoke_all_for_user` UBQ uses `conflicts:proceed` with no retry (`auth/sessions.py`);
  a version-conflicted session survives the logout-all meant to kill stolen sessions. Retry to
  zero (the reconcile pattern exists).
- **m-3** — disagreement recompute truncates at 10k findings/digest (`services/disagreement.py`);
  a >10k image computes flags from a partial set. Assert `total < size` or page.
- **m-4** — retention ages indices by client-supplied `@timestamp` (`last_seen_at`); a backdated
  scanner clock can age an index into early deletion. Consider a server-side append timestamp.
- **m-5** — one malformed `system-config` lifecycle-knobs doc raises out of the sweep, aborting
  rollover+retention for **all** clusters. Per-cluster catch → fall back to defaults.
- **m-6** — `system-audit-log` never rolls over (excluded from the lifecycle `SERIES`); retention
  deferral is documented, rollover deferral is not. One unbounded index until M9e.
- **m-7** — token-admin mint/rotate can't set `expiry`; every API-minted token is eternal until
  revoked (the `expiry` field is mapped + enforced but unreachable from the API).
- **m-8** — login-CSRF: `SameSite=Lax` on the response cookie doesn't stop a cross-site form from
  *issuing* the login POST (attacker logs a victim into an attacker account). Mutation APIs are
  safe (JSON forces preflight). Double-submit token on `/auth/login` or accept + document.
- **m-9** — admin user/role-management endpoints are unbuilt, so "role-change revokes live
  sessions" (an M5a DoD item) is unreachable and untested. In practice caps/`disabled`/`must_change`
  re-read per request, so it's belt-not-load-bearing — but the deliverable is silently unmet and
  not listed under Out-of-scope. Document the deferral or build it.
- **m-10** — "one session per browser" is claimed but not implemented; each login accretes a live
  session doc until TTL. Harmless (each revocable) but the claim is inaccurate.

## Nit

- **n-1** — `tenant_query` forwards caller `params` verbatim; a `q=` (Lucene) param or a `global`
  agg bucket could sidestep the `filter` context. Server-constructed only today, not user-reachable,
  and MVP is all-clusters-visible — but guard it when per-user `allowed_cluster_ids` grants land.
- **n-2** — decision lifecycle has no HTTP surface yet (M5c owns the router); confirm M5c registers
  those mutations in the RBAC/IDOR suite.

## Verified correct (worth stating)

Sessions (raw id never stored, domain-separated hash, server-side expiry/revocation, refresh on
mint). Oracle discipline (429 before any lookup; dummy-hash on unknown/non-local/no-hash; one
generic 401 everywhere; timing as flat as argon2 allows). Capability gate (`must_change` rejected
before the cap check; `"*"` honored in both the gate and risk-accept; unknown role fails closed).
Bootstrap admin true seed-once. Token rotate new-before-disable; hash never in any response. Audit
log append-only by construction; `revision = _version` a sound causal key. Tenant filter
structurally un-escapable via body. M4: legacy alias adoption correct (verified in git); write
index never dropped; conservative pre-rollover retention set + newest-data age; knobs in
`system-config` (not hardcoded); recompute clears on reconvergence AND on a dropped finding;
catalog reads sort by `scan_order` never `@timestamp`; per-scanner sanctity intact. All new M5a
knobs are in `CONFIGURATION.md`.
